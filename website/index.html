04 / TLS EXPLAINED

How TLS Works — The Handshake

Part of this lab required capturing and analysing the TLS handshake. Here is what each stage does and what I observed in Wireshark.

1
Client Hello
The browser sends supported TLS versions (1.2 / 1.3), a list of cipher suites it supports, and a random value. This is fully visible in Wireshark — it is not yet encrypted.
2
Server Hello
The server picks the TLS version and cipher suite from the client's list, and sends its own random value back. In my lab, the server selected TLSv1.3.
3
Certificate Exchange
The server sends its TLS certificate (nginx.crt). The browser can verify this was signed by a trusted CA. In my lab the cert was self-signed, so the browser showed a warning — which I proceeded through.
4
Key Exchange
Both sides derive the same session keys without ever sending the key over the network. This is the core of TLS security — even if you capture all the packets you cannot decrypt the session without the private key.
5
Encrypted Application Data
All HTTP content is now encrypted. In Wireshark the payload shows only as "Application Data" — you can see packet sizes and timing but not the actual HTML, cookies, or form data.
